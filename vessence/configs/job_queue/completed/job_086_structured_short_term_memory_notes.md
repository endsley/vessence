# Job #86: Structured Short-Term Memory Notes for Stage 3

Status: completed
Priority: 1
Created: 2026-04-29
Completed: 2026-04-29

## Objective

Replace the current freeform atomic short-term note writer with a structured recent-memory extraction pipeline so Stage 3 writes high-value, retrieval-friendly short-term memories instead of generic summaries.

## Context

Jane currently uses two different "recent context" mechanisms:

- **FIFO (`vault_web/recent_turns.py`)** for immediate conversational continuity inside a live session
- **Chroma short-term memory (`short_term_memory`)** for recent recall across sessions/hours

The system previously used thematic short-term summaries (`memory_type="short_term_theme"`) and `context_snapshot` rows. Those became polluted with protocol chatter and low-value summaries, so the live short-term collection was intentionally purged on **2026-04-29** and the active Stage 3 write path was switched to one atomic `memory_type="short_term"` note per turn.

Current active behavior:

- Stage 3 persistence path in `jane_web/jane_proxy.py` calls `ConversationManager.update_short_term_memory(...)`
- `update_short_term_memory(...)` in `memory/v1/conversation_manager.py` strips injected metadata and then writes one freeform summarized note to Chroma
- Retrieval in `memory/v1/memory_retrieval.py` now suppresses all legacy `short_term_theme` and `context_snapshot` rows

The current freeform atomic-note design is safer than the old thematic lane, but it is still too weak:

- it stores a single prose summary string instead of explicit facts
- it does not clearly separate durable facts from open loops
- it does not preserve structured artifacts like file paths, people, dates, tool actions, or pending work
- it does not adapt its output shape by turn type (code/debugging vs calendar vs messaging vs todos)

This job is to redesign short-term memory so the Stage 3 Chroma lane becomes materially useful again.

## Problem

The current short-term note writer still behaves like a summarizer. That is the wrong abstraction. Short-term memory should be a **retrieval note for the next few hours/days**, not a miniature transcript.

Bad short-term notes look like:

- generic conversation summaries
- assistant phrasing without concrete state
- broad prose with no nouns, files, dates, people, or action items
- notes that fail to distinguish "done" from "still pending"

Good short-term notes should preserve:

- exact decisions
- exact open loops
- exact entities and artifacts
- exact files/functions/classes when code was discussed
- exact appointments/dates/people/messages when personal tooling was discussed

## Desired Design

### 1. Structured extraction, not plain summarization

The writer should internally extract a structured object for each Stage 3 turn, with fields such as:

- `facts`: concrete recent facts worth recalling
- `decisions`: decisions or confirmed state changes
- `open_loops`: unresolved tasks, confirmations, next actions
- `artifacts`: file paths, function names, classes, APIs, tools, IDs, entities
- `people`: names of people or contacts involved
- `time_refs`: explicit dates/times/time windows
- `turn_kind`: e.g. `code`, `debugging`, `calendar`, `messages`, `todo`, `general`

This can be generated by the LLM or hybrid heuristics, but the stored memory should be shaped around explicit retrieval value rather than narrative prose.

### 2. Store a retrieval-friendly text plus metadata

Chroma still needs a text document to embed. The system should flatten the structured extraction into a compact retrieval note, for example short labeled bullets:

- `Facts: ...`
- `Decisions: ...`
- `Open loops: ...`
- `Artifacts: ...`
- `Time refs: ...`

Metadata should include as many structured fields as are practical for later filtering/ranking, for example:

- `memory_type="short_term"`
- `topic="turn_memory"`
- `turn_kind`
- `has_open_loop`
- `artifact_paths`
- `person_names`
- `tool_names`
- `session_id`
- `timestamp`
- `expires_at`
- `summary_style`

### 3. Use turn-type-specific extraction prompts

Do not use one generic extraction prompt for every turn.

At minimum, have different extraction behavior for:

- code/debugging turns
- scheduling/calendar turns
- messaging/email turns
- todo/task-management turns
- generic conversational turns

Examples:

- code/debugging should preserve file paths, symbols, errors, root causes, and next edits
- scheduling should preserve appointments, dates, times, and requested follow-up actions
- messaging should preserve sender/recipient names, requested actions, and unresolved message operations
- todo/task turns should preserve confirmed items, priorities, and remaining actions

### 4. Reject low-value writes

After extraction, skip the write entirely if the turn has no durable recall value.

Examples to skip:

- greetings
- tiny acknowledgments
- generic completions with no future relevance
- notes with no concrete entities, no decisions, and no open loops
- protocol/meta chatter

### 5. Distinguish resolved vs unresolved state

Open loops are often the most valuable short-term memory. The new format should explicitly preserve whether something is:

- completed
- requested but not done
- awaiting confirmation
- blocked
- partially implemented

This matters more than a generic summary.

### 6. Keep notes short and dense

The goal is **high information density**, not readability for humans.

Target:

- usually 1-5 short bullets
- no long paragraph summaries unless unavoidable
- prefer exact nouns over explanation

### 7. Do not reintroduce the old failure modes

The new design must not regress into:

- session-wide thematic clustering
- FIFO feedback loops
- raw protocol/data block contamination
- storing general "conversation summary" prose that adds no retrieval value

## Scope

### In scope

- redesign the Stage 3 short-term writer
- update prompts/heuristics for structured extraction
- update retrieval formatting/metadata handling if needed
- add tests for good vs bad short-term note generation
- update architecture docs to reflect the new short-term memory format

### Out of scope

- redesigning long-term memory archival
- changing FIFO as the immediate continuity mechanism
- bringing back `short_term_theme` retrieval
- bulk migration of old thematic rows (the collection was cleared on 2026-04-29)

## Implementation Outline

1. Update `ConversationManager.update_short_term_memory(...)` to generate a structured recent-memory extraction rather than a single freeform summary.
2. Add a helper that classifies the turn into a small set of `turn_kind` values.
3. Add turn-kind-specific extraction prompts or hybrid prompt templates.
4. Flatten the structured extraction into a compact retrieval note for Chroma.
5. Persist structured metadata fields alongside the flattened note.
6. Add a validation gate that skips notes with no concrete entities, no decisions, and no open loops.
7. Ensure metadata/protocol stripping still happens before extraction.
8. Update retrieval code if any metadata-aware ranking/filtering is needed.
9. Add targeted tests with realistic examples from:
   - code edits
   - calendar/appointment handling
   - messaging actions
   - todo/task confirmations
   - low-value chatter that should be skipped
10. Update docs so future sessions understand FIFO vs Chroma short-term roles.

## Verification

Implementation is complete when all of the following are true:

1. A fresh Stage 3 coding turn writes a short-term memory that preserves file paths, symbols changed, and any remaining open work.
2. A fresh Stage 3 calendar or messaging turn writes a short-term memory that preserves names/dates/actions rather than generic prose.
3. Low-value turns are skipped rather than stored.
4. No stored short-term note contains protocol/meta chatter.
5. Retrieval surfaces the useful structured notes for queries like:
   - "what were we fixing in Android?"
   - "what appointment did I ask about?"
   - "what file did we change?"
   - "what still needs to be done?"
6. Tests cover both positive cases and rejection cases.
