# Root Layout

Vessence now uses a three-root layout:

- Code root: `/home/chieh/vessence`
- Vault root: `/home/chieh/vault`
- Runtime data root: `/home/chieh/vessence-data`

This split exists so the software can be shipped cleanly without mixing code, personal files, and mutable runtime state into one directory.

## 1. Code Root: `~/vessence`

Purpose:
- Holds the application code, internal configs, startup scripts, service definitions, and software documentation.
- This is the part that should be portable, version-controlled, and safe to ship after repo cleanup.

What belongs here:
- Python packages and modules
- shell scripts and startup code
- Docker files and compose files
- internal architecture docs and specs
- tests
- static templates and example config files

What does not belong here:
- personal vault files
- API keys or live secrets
- logs
- Chroma/vector DB files
- runtime SQLite databases
- session state

Examples:
- `~/vessence/amber/`
- `~/vessence/jane/`
- `~/vessence/vault_web/`
- `~/vessence/configs/`
- `~/vessence/startup_code/`

## 2. Vault Root: `~/vault`

Purpose:
- Holds the user's personal file vault and knowledge artifacts.
- This is the human-facing content store: files Jane and Amber can browse, reference, organize, and serve back through tools.

What belongs here:
- markdown notes
- documents
- images
- audio
- PDFs
- archived personal knowledge files
- identity essays stored as user-owned content

What does not belong here:
- logs
- `.env`
- credentials
- runtime DBs
- temporary service state
- service crash output

Examples:
- `~/vault/documents/`
- `~/vault/images/`
- `~/vault/audio/`

Functional role:
- Amber and Jane read from this root when interacting with user files.
- Vault web serves files from this root.
- indexing tools treat this root as the authoritative file corpus.

## 3. Runtime Data Root: `~/vessence-data`

Purpose:
- Holds mutable runtime state required for the system to operate.
- This is the replacement for the non-vault portion of the old `~/ambient` root.

What belongs here:
- live `.env`
- credentials
- logs
- Chroma/vector memory stores
- runtime SQLite databases
- session files
- activity state
- bridge and service working state

Examples:
- `~/vessence-data/.env`
- `~/vessence-data/credentials/`
- `~/vessence-data/logs/`
- `~/vessence-data/vector_db/`
- `~/vessence-data/vault_web/vault_web.db`
- `~/vessence-data/user_state.json`
- `~/vessence-data/data/jane_sessions.json`

Functional role:
- services load secrets and environment from here
- Jane and Amber memory reads/writes resolve here
- logs and operational diagnostics are written here
- runtime web/session databases live here

## Why Three Roots Instead Of Two

Using only `~/vessence` and `~/vault` is not enough.

If runtime state lives inside `~/vessence`:
- the repo stops being clean and shippable
- secrets and local machine state bleed into the code tree

If runtime state lives inside `~/vault`:
- the personal vault gets polluted with service internals
- operational artifacts mix with user knowledge and files

The third root exists to separate:
- software
- personal content
- machine/runtime state

## Decision Rules

When deciding where something goes:

- If it is software logic or project documentation, put it in `~/vessence`.
- If it is a user file or knowledge artifact, put it in `~/vault`.
- If it is generated by running services or needed to keep services running, put it in `~/vessence-data`.

## Operational Contract

The intended environment model is:

- `VESSENCE_HOME=/home/chieh/vessence`
- `VAULT_HOME=/home/chieh/vault`
- `VESSENCE_DATA_HOME=/home/chieh/vessence-data`

`AMBIENT_HOME` may still exist temporarily as a backward-compatibility alias during migration, but it should no longer be treated as the long-term canonical root.
