Request Lifecycle

▶ End-to-End Message Flow classify → context → brain → persist

Every user message passes through the same pipeline, regardless of whether it's a greeting or a complex coding request. The pipeline runs inside stream_message() in jane_proxy.py.

Browser POST /api/jane/chat/stream | 1. Acknowledge receipt (SSE "ack" event) | 2. Intent Classification (gemma3:4b via Ollama, ~200ms) | Returns: level (greeting/simple/medium/hard), model hint, certainty % | Level maps to tier: greeting/simple → light, medium → medium, hard → heavy | 3. Session Summary Load | Reads existing session summary (topics, open loops) from disk | Format: JSON with up to 5 tracked topics, each capped at 200 chars | 4. Context Decision: skip_context or full build? | IF standing brain is alive AND turn_count > 0: | skip_context = True → lightweight safety context only | ELSE: | skip_context = False → full context build (memory, profile, task state) | 5. Context Assembly | skip_context path: session summary + last 6 turns (2400 chars) + user message | full build path: system prompt + ChromaDB memory + user profile + task state | 6. Brain Dispatch | Standing brain (preferred): write to stdin of long-lived CLI process | Fallback: spawn one-shot CLI subprocess | 7. Stream Response (SSE events: status, thought, delta, done) | 8. Persist Turns (async background thread) | Write to ChromaDB short-term memory + update session summary | 9. Emit "done" event to browser

Standing Brain Architecture

▶ Overview: Single Brain one CLI process, full conversation continuity

Jane runs a single, long-lived CLI process. All messages route to the same brain, ensuring full conversation continuity — no cross-brain context injection problems, no style discontinuity, no reasoning chain gaps.

ProviderDefault ModelEnv Override
ClaudeOpus 4.6JANE_MODEL_CLAUDE
GeminiPro 2.5JANE_MODEL_GEMINI
OpenAIo3JANE_MODEL_OPENAI

The model can be changed at runtime via the settings UI. Changing the model kills the current brain and respawns with the new model.

Key design choice: Context depth (slim vs. full memory retrieval) is determined locally by context_builder.py keyword matching, not by a separate classifier. This eliminates the ~200ms Ollama round-trip per message.
▶ Standing Brain Manager spawn, send, restart, reap

StandingBrainManager is a singleton that owns the single brain process. It handles its full lifecycle.

Process Spawn

Each brain is spawned as an asyncio.create_subprocess_exec() with stdin/stdout/stderr pipes. The working directory is set to $VESSENCE_HOME. Each process gets its own process group (start_new_session=True) for clean shutdown.

  • Claude: Uses --input-format stream-json --output-format stream-json. Waits for an init event on stdout that contains the session_id.
  • Gemini: Uses --approval-mode yolo --output-format text. Waits 2 seconds for process to stabilize.
  • OpenAI (Codex): Uses codex exec --approval-mode full-auto. Same 2-second wait.

Sending a Message

When send(tier, message, system_prompt) is called:

  1. Acquire the tier's async lock (prevents concurrent sends to the same brain)
  2. Check health: if dead, hit max failures (3), or hit max turns (20) → kill and respawn
  3. Build the prompt:
    • Turn 0 (first turn): "{system_prompt}\n\nUser: {message}\nJane:"
    • Turn 1+ (subsequent): Just message (raw)
  4. Write to stdin as NDJSON (Claude) or plain text (Gemini/OpenAI)
  5. Read response events from stdout, yielding (event_type, chunk) tuples
  6. Increment turn_count, reset failure counter on success

Forced Restart at 20 Turns

After MAX_TURNS_BEFORE_REFRESH = 20 turns, the brain is killed and respawned. This is because CLI processes accumulate context in their internal memory, and after enough turns the context window fills up and older information gets compressed or dropped. Restarting forces a clean slate with a fresh system prompt on the next turn.

Idle Reaper

A background task runs every 60 seconds and checks each brain's health:

  • Dead processes: Immediately respawned
  • CPU runaway: If CPU exceeds 15% for a sustained 1 hour AND the brain is idle (5+ min since last use), it gets killed and respawned. This prevents runaway processes from burning resources while protecting brains that are actively working.
Note: The reaper never kills a brain that is actively handling a request. The idle_seconds check ensures only truly idle brains are candidates for termination.
Key files:
jane/standing_brain.py — StandingBrainManager, BrainProcess, spawn/send/kill/reap
jane/brain_adapters.py — cold-start fallback when standing brain is unavailable
▶ Context Classification local keyword matching, drives context depth

Message complexity is classified locally by context_builder.py using keyword matching (~0ms). This drives context depth — greetings get slim context, complex tasks get full memory + task state. All messages route to the single standing brain.

Context & Memory Flow

▶ The skip_context Decision full build vs safety context

This is the most critical branching point in the entire system. Before assembling context, the proxy checks whether the target brain already has context from a previous turn:

_skip_context = ( brain is eligible for standing brain routing AND manager is started AND brain process is alive AND brain.turn_count > 0 ← has seen at least one message )

If _skip_context = True: the system assumes the brain's CLI process still remembers its system prompt and earlier conversation from its internal memory. It assembles only a lightweight "safety context" — a fallback in case the CLI's internal memory has drifted.

If _skip_context = False: full context build happens — memory retrieval from ChromaDB, user profile, task state, code map, everything.

What the safety context contains

  • Session summary: A gemma-generated paragraph tracking up to 5 conversation topics
  • Recent history: Last 6 turns from state.history, capped at 2400 characters total
  • The current user message

What the safety context does NOT contain

  • ChromaDB memory retrieval (long-term or short-term)
  • User profile or background
  • Task state or pending jobs
  • Essence personality (if active)
Known gap: On turn 2+ of any standing brain, ChromaDB is never queried. The brain relies entirely on its own CLI session memory for anything beyond the last 6 turns. If the CLI's context has compressed older messages, that information is lost.
Key files:
jane_web/jane_proxy.py — L828-868 (_skip_context decision and safety context assembly)

▶ Cross-Tier Context Passing how (and where) context gets lost

With a single brain, conversation continuity is straightforward. The brain maintains its own context window. Here's the detailed flow:

Scenario: User sends 5 messages

Msg 1: "hey" → classifier: greeting → light brain (haiku) Msg 2: "fix the login" → classifier: hard → heavy brain (opus) Msg 3: "what about X?" → classifier: medium → medium brain (sonnet) Msg 4: "yes, do it" → classifier: medium → medium brain (sonnet) Msg 5: "add tests too" → classifier: hard → heavy brain (opus)

What each brain's CLI memory contains

BrainCLI MemoryGaps
Light (haiku)System prompt + Msg 1 + responseKnows nothing about Msgs 2-5
Medium (sonnet)System prompt + Msg 3 + response + Msg 4 + responseKnows nothing about Msgs 1, 2, 5
Heavy (opus)System prompt + Msg 2 + response + Msg 5 + responseKnows nothing about Msgs 1, 3, 4. And between Msg 2 and Msg 5, it has a gap — doesn't know what sonnet discussed.

How the proxy tries to bridge this

The web server maintains a unified state.history list that contains ALL turns from ALL tiers. When _skip_context = True, the proxy injects a safety context assembled from this shared history:

safety_ctx = [Session summary] ← gemma-generated topic paragraph + [Recent exchanges] ← last 6 turns from state.history (all tiers) + User: {message} ← current user message This safety_ctx becomes the brain_message sent to the standing brain.

Why this is often insufficient

  • 6 turns / 2400 chars is very small. A single opus response can be 2000+ chars. After 2-3 exchanges, the older ones are truncated away.
  • No tier-switch detection. When the heavy brain receives Msg 5 after sonnet handled Msgs 3-4, the same thin safety context is used. There's no special "handoff" with richer context.
  • The brain's own CLI memory conflicts. Opus remembers Msg 2's context in detail (it's in the CLI's internal memory), but gets a compressed summary of Msgs 3-4. This creates an inconsistency: detailed old context + vague recent context.
  • No ChromaDB retrieval. Even though all turns are persisted to ChromaDB in the background, the skip_context path never queries ChromaDB. The memory that could bridge the gap is available but unused.
Known gap: Cross-tier amnesia. When tier switches happen, the new brain receives at most 2400 chars of recent history and a session summary. For conversations with rapid tier switching (e.g., casual greeting followed by a complex question), the complex brain can miss important context from the lighter brain's exchange.
Key files:
jane_web/jane_proxy.py — L843-868 (safety context assembly), L946 (brain_message selection)
jane/context_builder.py — _format_recent_history() (L558)
▶ Same-Tier Context Flow (Detail) what happens on turn 2+ of the same brain

When consecutive messages route to the same tier, the flow is simpler but still has nuances:

Turn 1 (brain.turn_count == 0)

  1. Full context build runs (PromptProfile, ChromaDB memory, user profile, etc.)
  2. JaneRequestContext is created with system_prompt (full) + transcript (user message)
  3. Standing brain's send() prepends system prompt to message: "{system_prompt}\n\nUser: {message}\nJane:"
  4. Brain receives its full identity, memory, and context

Turn 2-19 (brain.turn_count > 0, _skip_context = True)

  1. Safety context is assembled: session summary + last 6 turns + current message
  2. JaneRequestContext is created with system_prompt="" (empty) + transcript (safety context + message)
  3. Standing brain's send() sends just the message (no system prompt prefix, since turn_count > 0)
  4. Brain relies on its own CLI internal memory for system prompt, user profile, earlier turns
  5. The safety context provides a thin fallback: if the CLI's memory has compressed older turns, the session summary and recent history help fill the gap

Turn 20 (brain.turn_count == MAX_TURNS_BEFORE_REFRESH)

  1. Standing brain kills the process and respawns a fresh one
  2. turn_count resets to 0
  3. Next message triggers a full context build (turn 1 path)
  4. Brain gets a fresh system prompt with updated memory from ChromaDB
The 20-turn refresh exists because CLI processes accumulate context internally. After ~20 turns, the CLI's context window fills up and it starts compressing or dropping older messages. Restarting forces a clean slate with fresh ChromaDB memory, which may include facts learned during the 20 turns that the CLI's compressed context lost.
Key files:
jane/standing_brain.py — send() (L232-340), MAX_TURNS_BEFORE_REFRESH (L65)

Memory System

▶ ChromaDB Collections 4 collections, shared across all brains

All persistent memory is stored in ChromaDB vector databases. These collections are the shared substrate between the standing brain and Amber.

CollectionPurposeTTLShared With
short_term_memorySummarized conversation turns14 daysJane (all tiers), Amber
long_term_memoryPermanent facts, preferences, user informationNoneJane (all tiers), Amber
file_indexIndexed vault files with embeddingsNoneJane (all tiers), Amber
user_memoriesExplicitly saved facts via add_fact.pyNoneJane (all tiers), Amber

How short-term memory works

After each exchange, a background thread in _persist_turns_async() summarizes the turn using a local model and writes the summary to short_term_memory. The summary format is concise_turn_memory_v1 — typically compressing ~400 chars of raw conversation into ~190 chars of summary.

How memory retrieval works

The memory daemon (port 8083) receives a query (the user's message), performs cosine similarity search against ChromaDB embeddings, and returns the most relevant memories. The librarian then synthesizes these into a coherent paragraph that gets injected into the system prompt.

Known gap: Retrieval is semantic, not chronological. ChromaDB returns results by embedding similarity, not by recency. Asking "fix the installer" might not retrieve the turn where you discussed installer specs if the embeddings don't match well enough. The most recent conversation turn is not guaranteed to appear in retrieval results.
Key files:
jane/config.py — collection names, DB paths
agent_skills/local_vector_memory.py — ChromaDB read/write
agent_skills/janitor_memory.py — TTL cleanup, deduplication
startup_code/memory_daemon.py — FastAPI query server on port 8083
▶ Memory Write Path async persistence, race conditions

After each exchange, the proxy dispatches turn persistence to a background thread via _persist_turns_async(). This is non-blocking — the user sees the response immediately while persistence happens in parallel.

stream_message() completes | +--> background thread (daemon=True): | +--> conversation_manager.add_messages([user_turn, assistant_turn]) | Summarizes turn using local model | Writes summary to ChromaDB short_term_memory | Invalidates memory summary cache | +--> update_session_summary_async(session_id, user_msg, assistant_msg) Reads existing session summary Asks local model to update topics based on new exchange Writes updated summary to disk
Race condition: Because persistence is async, if the user sends a second message quickly, the second message's memory retrieval might not see the first message's response in ChromaDB yet. The background thread may still be writing. This is partially mitigated by the in-memory state.history (which is updated synchronously), but ChromaDB-based retrieval on a fresh brain won't see the latest turn.
Key files:
jane_web/jane_proxy.py — _persist_turns_async() (L532-570)
agent_skills/conversation_manager.py — add_messages(), turn summarization
jane/session_summary.py — update_session_summary_async()

Memory retrieval happens at different depths depending on the context path:

Turn 1, medium/hard levelYesYesYes
Turn 2+, any level (skip_context)NoYesYes (6 turns, 2400 chars)

On this path, ChromaDB is never consulted, even though it contains all recent conversation summaries. The brain depends entirely on its CLI's internal memory plus a thin 2400-char safety snippet. This is the primary reason context appears to be "forgotten" — the information exists in ChromaDB but is never read.

▶ Session Summary topic tracking, lightweight bridge

The session summary is a lightweight, on-disk JSON structure that tracks the current conversation's topics. It's the only piece of context that is both cheap to produce and always available on the skip_context path.

  • Max 5 topics tracked at a time, each capped at 200 chars
  • Updated after every exchange by a local model in a background thread
  • Thread-safe: Uses a write lock to prevent concurrent corruption
  • Format: JSON with topic name, state (open/resolved), and brief description

The session summary is the closest thing to a cross-brain shared memory that runs on every turn. However, it's a lossy compression — it captures topics, not details.

Key files:
jane/session_summary.py — load, update, format session summaries

Permission Broker (Claude Only)

When JANE_WEB_PERMISSIONS=1, Claude's standing brain runs with a PreToolUse hook that intercepts tool calls (Bash, Write, Edit). The hook POSTs to /api/jane/permission/request, and the broker routes the request to the browser via SSE. The user sees an approve/deny dialog. The emitter is registered under "standing_brain" so the broker can find it.

▶ Broadcast System multi-listener SSE, periodic summaries

When Jane is responding, any connected client (web browser, Android app) can subscribe to the broadcast stream. The StreamBroadcaster batches delta events and periodically publishes a summary of what Jane is saying so far.

  • Summary interval: Every few seconds, a local model generates a 1-sentence summary of the in-progress response
  • Multi-listener: Multiple browser tabs or devices can watch the same response in real-time
  • Cleanup: Listeners are automatically unsubscribed on disconnect

▶ Authentication Google OAuth, sessions, share codes

Multi-layer authentication protects all routes:

  • Google OAuth: Primary login via Google sign-in, token verified server-side. Restricted to allowed email addresses.
  • Session cookies: 30-day httponly cookies, SameSite=lax
  • Trusted devices: Separate cookie tracks known devices
  • Localhost bypass: Requests from 127.0.0.1 skip auth for local access

Essence System

▶ Essence Loader & Activation manifest.json, personality swap, memory isolation

Essences are self-contained AI agent packages. Each lives in a folder under essences/ with a manifest.json describing its name, personality, tools, and ChromaDB path.

  • Auto-load: At startup, the loader scans the directory and registers all valid essences
  • Memory isolation: Each essence has its own ChromaDB collection. Memories don't bleed between personas.
  • Personality injection: When activated, the essence's personality replaces Jane's base system prompt
  • Universal memory: Jane's core long-term memory remains accessible during essence activation
  • Hot-reload: New essences can be added without restarting the server

Infrastructure

▶ Docker & Deployment slim images, first-boot install, Cloudflare tunnel

Known Gaps & Planned Fixes

▶ Context Continuity Gaps where and why memory is lost

1. Cross-Tier Amnesia

When messages route to different tiers, the receiving brain has no knowledge of what other brains discussed. The safety context provides at most 6 turns / 2400 chars of shared history, which is often insufficient for complex conversations.

Impact: "I just told haiku about X, now opus doesn't know about it."

2. skip_context Bypasses ChromaDB

On turn 2+ of any standing brain, ChromaDB is never queried. This is the majority of all messages. Even though conversation summaries exist in short-term memory, they are never read on this path.

Impact: The brain relies entirely on its CLI's internal memory, which degrades over many turns.

3. Safety Context is Too Thin

6 turns / 2400 characters is approximately one detailed exchange. For conversations with back-and-forth, this window is exhausted quickly. There's no tier-awareness — the same thin context is used whether the tier just switched or not.

Impact: After rapid exchanges, earlier context falls out of the safety window.

4. Async Persistence Race

Turn persistence runs in a background thread. If two messages arrive in quick succession, the second message's context build may not see the first message in ChromaDB yet.

Impact: Rare in practice (users don't usually send messages faster than the ~500ms write latency), but possible.

5. No Tier-Switch Detection

The proxy doesn't track which tier handled the previous message. When a tier switch happens, no special context handoff occurs. The new brain gets the same thin safety context as if it were a same-tier continuation.

Impact: Tier switches (common in natural conversation) have no extra protection against context loss.

Planned Mitigation

  • Add memory retrieval to skip_context path: Query ChromaDB even on turn 2+. The query is ~100ms — a small cost for significant reliability improvement.
  • Tier-switch detection: Track state.last_tier. When tier changes, force full context build instead of skip_context.
  • Increase safety context window: Tier-aware caps (light: 4 turns/1600 chars, medium: 10 turns/6000 chars, heavy: 15 turns/12000 chars).
  • Treat brains as stateless with a bonus cache: Always inject enough context that the brain can respond correctly even if its CLI memory were completely empty.
